# CMS-214 Studio Runtime Bundle Analysis

Date: 2026-05-01

## Scope

This records the current Studio runtime bundle composition before any code
splitting work. The goal was to determine whether the 13 MB runtime is caused
by route-level UI size, TipTap/editor dependencies, missing production
optimization, or accidental server-side dependencies in the browser bundle.

## Root Cause

The primary size issue is not TipTap route weight. The browser runtime imports
`createMdxAutoFormFields` from the `@mdcms/shared/mdx` barrel:

- `packages/studio/src/lib/remote-studio-app.tsx`
- `packages/studio/src/lib/mdx-props-editor-host.tsx`
- `packages/studio/src/lib/runtime-ui/components/editor/mdx-component-catalog.ts`

That barrel re-exports `packages/shared/src/lib/mdx/extracted-props.ts`, which
imports `typescript` and `node:path` for server-side/host-side MDX prop
extraction. Because the runtime bundle consumes the barrel, Bun includes the
TypeScript compiler and Node shims in the browser runtime.

The production bundle still contains TypeScript APIs such as `createProgram`,
`readConfigFile`, and `parseJsonConfigFileContent`, confirming this is live
bundle content rather than only sourcemap metadata.

## Secondary Findings

- The checked-in `buildStudioRuntimeArtifacts` pipeline uses `minify: false`.
  That alone keeps the JS entry at about 13.7 MB instead of 5.85 MB.
- A production-mode build switches React and React DOM from development CJS
  sources to production sources.
- Externalizing TipTap/ProseMirror in production only reduced the estimate from
  5.85 MB to 5.40 MB. TipTap is meaningful but not the dominant first fix.
- Externalizing Highlight.js/Lowlight reduced the estimate from 5.85 MB to
  5.67 MB.
- Externalizing Zod reduced the estimate from 5.85 MB to 5.58 MB.
- Externalizing React/React DOM only reduced the estimate from 5.85 MB to
  5.66 MB, and it is not directly usable with the current backend-served
  browser module because bare imports would need host import-map support.

## Recommended Fix Order

1. Split the browser-safe MDX auto-form helper away from the Node-side
   TypeScript extractor import path. The clean package-level shape is a
   dedicated export such as `@mdcms/shared/mdx/auto-form`, then update Studio
   runtime imports to use that path. This should remove the TypeScript compiler
   and Node shims from the runtime.
2. Change `buildStudioRuntimeArtifacts` to emit a production/minified browser
   bundle. This should use production React sources and minification in the
   actual checked-in build pipeline, not only ad hoc CLI builds.
3. Add a runtime bundle size regression test or analyzer test that fails if the
   built runtime contains the TypeScript compiler or exceeds a target raw size.
4. Only after those fixes, re-evaluate code splitting. The first two changes
   are enough to estimate a production runtime around 1.64 MB raw before route
   splitting.

## Caveats

The `--external` builds are estimates. They prove relative dependency weight,
but directly externalizing `typescript` would leave an invalid browser import.
The real fix is to remove the import path that reaches `typescript`, not to mark
it external in the runtime build.

## Implemented Result

The implemented fix follows the estimate without externalizing browser imports:

- `@mdcms/shared` now exposes `@mdcms/shared/mdx/auto-form` as a browser-safe
  subpath that does not re-export the Node-side TypeScript extractor.
- Studio runtime imports now use that subpath for MDX auto-form helpers.
- `buildStudioRuntimeArtifacts` now minifies runtime JavaScript and inlines
  `process.env.NODE_ENV` as `"production"`.
